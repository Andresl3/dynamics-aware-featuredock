Dynamics-aware FeatureDock

A docking model that reasons about pockets that move. We add a per-residue µs–ms conformational-exchange signal to FeatureDock's binding-site representation and test whether it improves ligand-pose prediction.

Transformer docking Dyna-1 µs–ms dynamics 6×80 → 6×81 tensor PyTorch
Installation →

#Overview

Take FeatureDock — a transformer that predicts where a ligand sits in a protein pocket — and make it dynamics-aware: give it not just the static shape of the pocket but a readout of which residues move, so it docks more reliably in flexible pockets where a single crystal structure misleads it.

FeatureDock reads one static structure and predicts a per-grid-point “ligand-occupancy envelope” over the pocket. That works when the pocket is rigid; it struggles when the pocket breathes on the µs–ms timescale — exactly the motion a crystal structure cannot show. We test the hypothesis that telling the model which residues are conformationally mobile lets it resolve those cases.

#Architecture

One extra channel, threaded through the existing model. Deliberately minimal, so any change in performance is attributable to the dynamics signal rather than a bigger network.

Each binding-site voxel is described by 6 shells × 80 properties. We append the Dyna-1 predicted µs–ms exchange probability of the nearest residue as an 81st property per shell, making the tensor 6 × 81, then train the transformer. Both arms — baseline (6×80) and Dyna-1 (6×81) — are otherwise identical.

Warm-start

Both arms can also be fine-tuned from FeatureDock's pretrained 20-block checkpoint. The baseline loads all 334 weight tensors; the Dyna-1 arm loads 330 and re-initializes the 4 input-normalization tensors that must resize for the extra channel — isolating the dynamics channel as the only substantive difference.

#Results Measured

Both arms trained identically on PDBBind v2020 refined (train 4128 / val 353), warm-started from the pretrained checkpoint. Pose RMSD to the native crystal ligand uses FeatureDock's protocol: 500 random rotations, best of the top-4 scored poses, 2 Å success threshold.

Headline: the Dyna-1 dynamics channel lowers ligand-pose RMSD on all six tested targets and raises the 2 Å success rate on five of six. Pooled over 120 structures, near-native successes rise from 51 → 67.

On the structure

The clearest gains are cases where the baseline top-4 pose is several ångström off and the Dyna-1 channel pulls it onto native: PTP1B 5.1 Å → 0.4 Å, HSP90α 7.8 Å → 0.5 Å.

On the binding-site envelope-reconstruction task (the training objective), the two arms are statistically indistinguishable on held-out CDK2 (all metrics |Δ| ≤ 0.007). The dynamics channel does not change how the model draws the envelope — it changes which poses the scoring stage selects, which is where the RMSD improvement comes from.

#Installation

A SLURM cluster with a V100 (32 GB) GPU is the target. Featurization and training use the GPU; posing/eval are CPU-bound.

1. Clone and build the environment

git clone https://github.com/natesana/dynamics-aware-featuredock.git
cd dynamics-aware-featuredock
bash setup_hackathon_mamba.sh      # clones the 3 tool repos + builds the conda env

setup_hackathon_mamba.sh clones featuredock/, Dyna-1/ and protpardelle-1c/ into the repo root (they are gitignored, never committed) and creates a Python 3.11 conda env. Note the env name it makes — every SLURM script has an ENV_NAME="dynafeat" line near the top; set it to your env name. Each script also uses absolute cluster paths, so edit ROOT / OUT_DIR at the top of each script to your own scratch path before submitting.

2. Get PDBBind v2020 (refined set)

Register (free) at pdbbind.org.cn and download the refined set. Untar it and point the pipeline at it:

export PDBBIND_DIR=/your/path/to/PDBbind_v2020_refined/refined-set

3. Get the Dyna-1 channel

This is the one expensive precompute. If a teammate has the precomputed CSVs (dyna1_csv.tar.gz, ~16 MB), use them and skip the GPU step:

mkdir -p $OUT_DIR/dyna1_csv
tar -xzf dyna1_csv.tar.gz -C $OUT_DIR/dyna1_csv   # one CSV per PDB: position,residue,p_exchange

Or regenerate them yourself (GPU, hours):

sbatch hpc/run_dyna1_precompute.slurm             # Dyna-1 inference over all refined structures

#Using the model

Featurize once, then train and evaluate both arms. Every step is a SLURM script you submit; the full runbook with gotchas is in docs/REPRODUCE.md.

  1. Preprocess → feature tensors (GPU). Builds the baseline and augmented tensors side by side. 
    sbatch hpc/run_preprocess.slurm
# -> $OUT_DIR/pvar_80/  (baseline 6x80)   $OUT_DIR/pvar_81/  (Dyna-1 6x81)
# -> voxels/, het/{pid}_ligand.sdf, and occupancy labels
  2. Split + train both arms (GPU). The split holds out CDK2's clan as the test set (train=4128, val=353, test=18); training runs 20 epochs (~1.2 h/arm on a V100). 
    sbatch hpc/run_train.slurm                # from-scratch, both arms
sbatch hpc/run_train_warmstart.slurm      # optional: fine-tune from pretrained checkpoint
    Weights land in results/baseline80/, results/dyna1_81/ (and the *_warm/ variants).
  3. Evaluate binding-site occupancy (fast, one forward pass per structure). 
    sbatch hpc/run_evaluate.slurm             # CDK2 head-to-head -> cdk2_summary.csv
  4. Evaluate pose RMSD (the primary metric; 500 rotations, top-4 scored + oracle). 
    sbatch hpc/run_rmsd_full18.slurm          # 18 CDK2 structures
sbatch hpc/run_rmsd_screen.slurm          # the six-target screen, n=20/target
    evaluate_cdk2_rmsd.py reports per structure: top1 (best-scored pose), top4 (best RMSD among the top-4 scored — the paper-comparable number), and oracle (min RMSD over all restarts — a best-case diagnostic).
Just want to test, not retrain? Drop the trained .torch checkpoints into $OUT_DIR/results/<name>/ with the filenames in docs/HOW_TO_GET_WEIGHTS.md, then run the eval scripts directly — no training needed.

Common gotchas

  • conda in batch jobs: a non-interactive SLURM shell has no conda function. Every script sources etc/profile.d/conda.sh first — don't remove that block.
  • logs off scratch: #SBATCH --output/--error point at $HOME/dyna_logs; writing logs onto Lustre scratch caused intermittent Errno 116 stale file handle crashes.
  • each --steps is a full epoch over all 4128 structures, not a minibatch. --steps 20 = 20 epochs.
  • warm-start memory: both arms must be built --n_blocks 20 to match the checkpoint; lower --n_structs to 4 and set PYTORCH_CUDA_ALLOC_CONF=expandable_segments:True to fit 32 GB.

#Built on

ToolRole
FeatureDock The base docking model, rewritten & retrained with the extra channel
Dyna-1 Produces the per-residue µs–ms conformational-exchange channel
Protpardelle-1c Generates conformer ensembles (for the ensemble-scoring extensions)

Training data: PDBBind v2020 refined (~4,515 preprocessed cocrystals), with the paper's clan-based split so homologs never straddle train/test. Primary metric: ligand-pose RMSD to native (2 Å success threshold).
